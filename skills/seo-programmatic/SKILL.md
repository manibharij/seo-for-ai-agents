# Programmatic SEO — scale, without spam

A specialist skill for generating many pages from a dataset — done **white-hat**. Programmatic SEO is legitimate and powerful (think genuinely useful location pages, product pages, comparisons, data-driven tools). It is also the single easiest way to wreck a site, because the lazy version is exactly what the rest of this pack fights: **thin, near-duplicate, doorway pages mass-produced to chase keywords.** This skill does the real version and **refuses** the spam version.

> The line, stated up front: **every generated page must offer genuine, distinct value a user would want — or it shouldn't exist.** If pages differ only by a swapped variable with no real per-page substance, that's a doorway-page pattern; the answer is fewer, better pages (or none), not thousands of thin ones. The agent will say so rather than generate spam.

Work the four steps: **Assess the dataset → Design → Generate (with the quality gate) → Verify.** This interacts heavily with Reach (crawl budget), Read/content-audit (thinness), Connect (linking), and Understand (schema).

---

## Step 1 — Assess the dataset & intent

Before generating anything:
- **Is there real per-page substance?** Does the dataset give each page genuinely distinct, useful content (real data, real differences), or would pages be near-identical templates? This is the go/no-go.
- **Is there real demand & intent?** Do people actually search for these page types? (With a connected keyword tool, size it; without, reason from the domain and say so.) Don't generate pages for queries no one makes.
- **What's the honest page count?** Often far fewer than hoped — only the entities with enough real data to support a worthwhile page qualify. Be honest about that number.

If the dataset can't support distinct, useful pages, **stop and say so** — recommend fewer high-quality pages, enriching the data first, or not doing it. That recommendation *is* the right output here.

## Step 2 — Design

- **Template + unique substance.** Design a template that frames consistent structure, but ensure each page is filled with **genuinely page-specific** content from the data (and, where possible, unique elements: real local detail, real stats, real specifics) — not boilerplate with a name swapped.
- **Intent-matched layout.** Lead with the answer/value the searcher wants (ties to Read/Cite).
- **Honest schema.** Plan the right type per page (`Product`, `LocalBusiness`, etc.) populated from real data (rung 3 rules — only true values).
- **Internal linking & discovery.** Plan hub/category pages and cross-links so the set is reachable and not orphaned (rung 4), plus sitemap generation.
- **Indexation/crawl-budget plan.** At scale, decide what should be indexable vs `noindex`/canonicalised, so you don't flood crawl budget with low-value combinations (rung 1; see `references/programmatic-quality-bar.md`).

## Step 3 — Generate, behind a quality gate

- Generate from the real data, applying a **quality threshold**: a page is only published if it meets a minimum bar of genuine, distinct content (e.g. enough real data fields, a real description, unique value) — **pages below the bar are not published** (or are consolidated). This gate is what separates pSEO from doorway spam.
- **Do not auto-write fake substance** to pad pages to the bar. If data is missing, the page doesn't qualify (or a human supplies real content). Same white-hat line as content-editing: never fabricate.
- Keep it in version control; on an existing site, roll out in batches and watch (existing-site safety).

## Step 4 — Verify (on the served output)

- Sample generated pages across the set and confirm on the **served HTML**: real, distinct, substantive content (not thin/near-duplicate), correct and honest schema, intent match, and that they're reachable/linked (not orphaned) and in the sitemap.
- **Duplication check:** sampled pages aren't near-identical to each other (run a few side by side). If they are, the gate failed — raise the bar or cut pages.
- **Crawl-budget check:** the indexable set is the valuable subset; low-value combinations are `noindex`/canonicalised, not flooding the index.
- Report honestly which pages passed the bar and how many were withheld.

---

## The white-hat lines (non-negotiable)
- **No doorway pages** — near-duplicate pages whose only differences are swapped keywords/locations with no real per-page value.
- **No fabricated substance** to inflate thin pages to the quality bar.
- **No mass-generation for the index's sake** — quality and genuine usefulness gate quantity, always.
- When the honest answer is "this dataset can't support good pages at scale," say it. Recommending *fewer, better* pages is a valid, often correct outcome.
